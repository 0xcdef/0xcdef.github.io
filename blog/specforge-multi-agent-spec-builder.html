Build specs that are actually ready to execute.

SpecForge is a Claude skill that runs your requirements through two rounds of adversarial multi-agent audit before producing an executive plan — catching ambiguity, gaps, and feasibility risks before a single line of code is written.

Methodology · Apr 12, 2026 · 8 min read

The problem with specs written by one agent

Ask a single LLM to write a design specification and it will produce something that looks thorough. The sections are there, the terminology is correct, the structure is clean. But it will also quietly skip the things it doesn't know how to answer, resolve ambiguous requirements in whichever direction felt natural, and assert that third-party APIs work exactly the way it remembers — which may be a version or two out of date.

The spec looks done. It isn't. And the gap between looking-done and actually-done is where projects go wrong.

"Audit agents are adversaries, not assistants. Their job is to find problems. The master's job is to synthesize solutions."

SpecForge is a Claude Code skill built around a simple fix: after the master agent writes a draft spec, it dispatches three independent audit agents — each with a single, adversarial job. One checks completeness. One hunts for ambiguity. One stress-tests feasibility. They report back. The master triages, applies the findings, and only then hands a release spec to the user for confirmation. The same process repeats on the executive plan.

The workflow in seven steps

SpecForge runs through a fixed sequence of phases separated by hard gates. No phase can be skipped. No gate auto-confirms.

Phase 0 — Requirements elicitation

The master conducts a structured six-question interview: what is it, who uses it, what are the core goals, what are the constraints, what is out of scope, and what does success look like. If the user's first message already answers most of these, the master extracts them directly and only asks what's missing. Output is requirements.md — the source of truth that every subsequent artifact traces back to.

Gate 0 holds here. Nothing moves until the user confirms the requirements are correct.

Phase 1 — Domain research + draft spec

Before writing, the master probes for available research tools — AutoStudy KB first, deep search skill second, uploaded reference documents third, agent reasoning only as the last resort. Whatever is used gets declared in the spec header as a research basis. This matters: the Auditor-Feasibility reads that field and raises its critical threshold when the basis is agent reasoning only, because unsupported API assumptions become Critical findings rather than Major ones.

Phase 2 — Audit Round 1: three lenses

Three subagent instances run one-pass against the draft spec, each with a different adversarial system prompt:

Lens 1

Auditor-Completeness

Does the spec address every requirement and success criterion? Flags gaps only — no suggestions, no rewrites.

Lens 2

Auditor-Ambiguity

Finds every sentence that two engineers could interpret differently. Quotes the exact text, names both interpretations, rates which is more dangerous.

Lens 3

Auditor-Feasibility

Stress-tests implementation paths, API assumptions, scale assumptions, and dependency risks. Escalates to Critical when research basis is agent-only.

Auditors never see each other's output. Separation is the point — overlapping findings signal high importance; divergent findings broaden coverage.

Phase 3 — Master synthesis

The master reads all three reports and applies a deterministic triage algorithm: classify each finding as Critical, Major, or Minor; deduplicate cross-auditor overlaps; apply all Critical and Major findings; log Minor findings without acting on them; write a rejection reason for anything deliberately not applied. Every finding gets an explicit disposition in audit_log_r1.md — nothing is silently ignored.

Gate 1 holds after spec_v1.md is delivered. The user confirms or requests amendments before the executive plan begins.

Phases 4–6 — Executive plan + second audit

The executive spec transforms the confirmed design spec into a phased build plan: phases with deliverables and testable acceptance criteria, a dependency graph with the critical path identified, interface contracts, a risk register, and an open-questions table with owners. Before writing any phase breakdown, the master resolves or explicitly assigns all open questions to specific phases — no OQ is deferred silently into the risk register.

Two more audit lenses then review the executive plan: Auditor-Consistency (does it faithfully implement the release spec?) and Auditor-Executability (can it be handed to an agent and run without further architectural decisions?). The master applies a second triage pass and delivers exec_spec_v1.md.

Gate 2 is the final hold. Unresolved open questions block this gate — every OQ must be closed or explicitly deferred to a named post-v1 milestone with the user's acknowledgement.

What the audit actually catches

During the test run that produced this post, SpecForge processed a SystemVerilog dependency parser project from a single sentence of requirements to a full executive plan. Thirteen findings surfaced across the two audit rounds. A few are worth highlighting because they represent the exact class of bugs that kill projects quietly.

The ambiguity that breaks downstream tools

The draft spec said that unresolved dependencies would be logged to an unresolved[] array. Auditor-Ambiguity flagged this as Critical: it was unclear whether all types of unresolved references — modules, includes, packages, interfaces — went into one flat array, or whether each type was tracked separately. Two engineers would implement this differently. The fix was a typed dict: unresolved{modules, includes, packages, interfaces}. A downstream tool consuming this JSON would behave correctly with one interpretation and silently lose data with the other.

The feasibility risk hiding in plain sight

The spec assumed pyslang's Python API could extract module instances, package imports, and interface usage with straightforward AST walking. Auditor-Feasibility rated this Critical because the research basis was agent reasoning only — the pyslang API surface for structural extraction is non-trivial and version-dependent. The fix: pin to pyslang >= 6.0, add an open question requiring an empirical API spike on day one of that phase, and document a regex fallback path if the API proves insufficient.

The executability gap that stalls implementation

The executive spec's Phase 4 said to resolve `include directives via the incdir chain — but didn't specify the resolution order. File-relative first then incdirs? Incdirs only? Incdirs first? Auditor-Executability rated this Critical: an agent starting that phase would have to make an architectural decision not in the spec. The release spec had specified it in Design Decision D6; the executive spec had just failed to restate it. One line fixed it.

Degraded mode and explicit invocation

SpecForge is designed for use with real subagent dispatch — claude -p one-pass calls or Cohole crew dispatch. When subagents aren't available, the skill runs in degraded mode: the master switches audit personas sequentially and explicitly. But before doing so, it stops and tells the user: the same model auditing its own output has blind spots, and the user should confirm they want to proceed. This notice is not optional — silent degradation is how quality loss goes unnoticed.

The skill is also explicitly invocation-only. It will not activate because you mentioned a spec, a plan, or requirements in passing. You invoke it with /spec-forge, use spec-forge, or activate spec-forge. This was a deliberate design decision: SpecForge runs a heavyweight multi-phase workflow. It should never fire unexpectedly.

The artifact trail

Every SpecForge run produces a traceable artifact chain. Each file has a status that advances through pending → draft → confirmed → final. The registry doubles as a session handover block — if the session is interrupted mid-run, the next session can read the registry and resume from the correct phase.

Getting the skill

SpecForge ships as a single SKILL.md file dropped into your Claude skills folder. It works with Claude Code and any setup that supports the skills system. The full file — including all auditor system prompts, the triage algorithm, gate language, and the artifact registry — is available in the Cohole skills collection.

The test project used to validate the skill — the SystemVerilog dependency parser — is a real tool worth building. If you want to see the full requirements.md, spec_v1.md, and exec_spec_final.md that came out of the run, those are included alongside the skill.
