Build and ship a GitHub Pages site entirely through Claude.

How to connect Claude Desktop to GitHub via MCP, create a Jekyll site, design it iteratively, and push blog posts — all from a single conversation.

Tooling & Workflow · Mar 2026 · 8 min read

Why this matters

GitHub Pages gives you a free, fast, version-controlled website. Jekyll turns Markdown into a real site. Claude, connected to GitHub via MCP, can read, write, and push files on your behalf — making the entire workflow conversational. No terminal required for day-to-day publishing.

This post documents the exact steps: connecting Claude Desktop to GitHub, scaffolding the site, choosing and fixing a theme, and establishing a repeatable blog-post workflow.

"The best developer tool is the one you actually use. A conversation is the lowest-friction interface that exists."

Step 1 — Connect Claude Desktop to GitHub via MCP

Claude Desktop supports the Model Context Protocol (MCP), which lets it talk to external services through a local server process. GitHub's official MCP server exposes your repos as tools Claude can call directly.

Install the MCP server

You need Node.js installed. Verify with:

# Check Node is available
node --version
npx --version

Then edit Claude Desktop's config file:

# macOS
~/Library/Application Support/Claude/claude_desktop_config.json

# Windows
%APPDATA%\Claude\claude_desktop_config.json

Add the GitHub MCP server block:

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_yourTokenHere"
      }
    }
  }
}

Create a GitHub Personal Access Token

Go to GitHub → Settings → Developer Settings → Personal Access Tokens. Generate a fine-grained token with repo scope (read and write access to your target repository). Paste it into the config above, then fully restart Claude Desktop.

Verify the connection

Start a new Claude conversation and ask: "List files in my repo X." If Claude returns a file listing, the MCP server is live. If you see errors, check that npx is in your PATH and that the token has the correct scopes.

Step 2 — Scaffold a GitHub Pages site

GitHub Pages natively supports Jekyll. The minimum viable setup is three files: _config.yml, Gemfile, and index.markdown. Claude can create all three in a single push.

_config.yml

Site metadata & theme

Title, description, author links, and the remote_theme directive that tells GitHub Pages which Jekyll theme to use.

✓ Use remote_theme not theme for third-party themes
Gemfile

Ruby dependencies

Use the github-pages gem to pin all dependency versions to what GitHub Pages actually runs. Add jekyll-remote-theme as a plugin.

✓ Delete Gemfile.lock after any theme change

A working _config.yml for the Minimal theme looks like this:

title: "Your Site Title"
description: "One line describing your focus."
github_username: "yourusername"
twitter_username: "yourhandle"

remote_theme: pages-themes/minimal@v0.2.0
plugins:
  - jekyll-feed
  - jekyll-remote-theme

Once pushed, GitHub Actions will build the site automatically. Check Settings → Pages in your repo to confirm it is deploying from the main branch.

Step 3 — Choose and fix a theme

GitHub Pages supports a small list of built-in themes and any public GitHub repo via remote_theme. The two most useful for technical blogs are:

ThemeLayoutBest forGotcha
Minimal Sidebar + content Personal profiles, blogs Override header via _layouts/default.html
Slate Full-width Docs, single-page projects Native bg is #212121 — match your CSS to it
Cayman Centered hero Project landing pages Limited customisation without layout override

The most common theme bug

Mixing layout: home (a Minima layout) with a non-Minima theme causes Jekyll to silently fall back to the theme's README. Always use layout: default as your starting point, then override from there.

Overriding the header

Create _layouts/default.html in your repo. Jekyll checks your repo before the theme gem, so your file wins. This is the cleanest way to replace a generic site title with a proper profile header — name, tagline, bio, social links, and topic tags — all styled to match the rest of the site.

Step 4 — Style with a custom SCSS file

Create assets/css/style.scss with a Jekyll front-matter block at the top. GitHub Pages compiles it and serves it as style.css.

--- (front matter — required)
---

/* Override theme SCSS variables BEFORE the import */
$body-background: #1a1a1a;
$body-link-color: #4ecdc4;

@import "{{ site.theme }}";

/* Your custom rules after the import */
:root {
  --teal: #4ecdc4;
  --coral: #ff6b6b;
}

Variables set before the @import override the theme's defaults at compile time. Rules written after the import use normal CSS specificity. Use !important sparingly — only when the theme applies inline styles you can't otherwise override.

Key CSS patterns that make dark-theme articles readable on mobile:

/* Fluid type — no JS, no breakpoints */
font-size: clamp(28px, 6vw, 58px);

/* Horizontal scroll for wide blocks */
overflow-x: auto;
-webkit-overflow-scrolling: touch;

/* Hide sidebar entirely on mobile */
@media (max-width: 780px) {
  .sidebar { display: none; }
}

Step 5 — Create a repeatable blog post workflow

Store a template at blog/_template.html. Each new post is a copy of that file with content filled in. Because the file lives in your repo, Claude can read it at the start of any session and follow the exact style — palette, component classes, sidebar structure — without needing to be re-briefed.

The template design system

The established palette for this site uses a warm parchment article surface (#F7F5F0) against a dark ink masthead and footer (#1A1916), with three accent colours:

Terracotta #C4572A

Primary accent

Used for h3 subheadings, category badges, and the pull-quote left border. Signals structure.

Teal #1D7A6A

Link & positive colour

All body links, diagram step highlights, and "good" states in comparison tables.

Also: amber #E8A060 for hero accents

Updating the index page

After publishing a new post, add a .post-item block to index.markdown. Each item has a category tag, date, title link, one-line description, and a "Read the post →" link. Claude can push both files in a single commit.

<!-- index.markdown — post-item pattern -->
<div class="post-item">
  <div class="post-item-meta">
    <span class="post-item-tag">Category</span>
    <span class="post-date">Mar 22, 2026</span>
  </div>
  <a class="post-item-title" href="/blog/your-post.html">Post title</a>
  <p class="post-item-desc">One sentence description.</p>
  <a class="post-item-link" href="/blog/your-post.html">Read the post &rarr;</a>
</div>

Lessons learned

A few non-obvious things that cost time during this build:

Stale Gemfile.lock kills builds silently. When you change themes or add plugins, delete Gemfile.lock entirely. GitHub Pages regenerates it. Keeping an old lock file causes the build to fail with no obvious error message in the Pages UI.

CSS cascade order matters more than specificity. Variables set before @import in style.scss modify the theme at compile time — this is more reliable than trying to override with higher-specificity selectors after the fact. Always use the SCSS variable approach for colours and backgrounds.

Mobile sidebars overlap rather than reflow. A CSS grid with position: sticky on the sidebar does not collapse gracefully — it escapes the grid flow and overlaps the article on narrow screens. The correct fix is display: none at your mobile breakpoint, not order: -1.

The template file is your memory. Claude has no persistent memory between sessions. Storing the blog template in the repo at blog/_template.html means you can start a new session with "read my template and create a post about X" and get consistent output every time.

Putting it all together

The full workflow, once set up, fits in a single conversation: tell Claude what you want to write, it reads the template, produces the HTML, pushes the blog file and the updated index in one commit, and GitHub Pages deploys automatically within two minutes.

For an engineering-focused site, this setup hits a good balance — the blog posts are static, fast, and version-controlled; the design is consistent without a build pipeline; and the publishing workflow has zero friction beyond the writing itself.
