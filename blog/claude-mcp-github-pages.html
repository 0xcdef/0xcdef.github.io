Building a GitHub Pages Site with Claude & MCP

How to wire Claude Desktop to your GitHub repo, let an AI agent push code, design a Jekyll site, and iterate on UI — all through conversation.

AI & Tooling · March 2026 · 8 min read

What is MCP and why does it matter?

Model Context Protocol (MCP) is an open standard that lets AI assistants like Claude talk directly to external tools — file systems, APIs, databases, and services like GitHub — without you copying and pasting content back and forth. Instead of describing what you want and manually applying the result, Claude can act: read files, write commits, push branches, and open pull requests on your behalf.

“MCP turns Claude from a text generator into an autonomous agent that can operate your tools.”

For developers, this changes the workflow fundamentally. Your repository becomes a shared workspace between you and Claude. You describe intent; Claude handles the mechanical execution. The conversation is the version history.

Step 1 — Connect Claude Desktop to GitHub

Claude Desktop supports MCP servers configured in a local JSON file. GitHub provides an official MCP server via npx, so no separate installation is required beyond Node.js.

Create a GitHub Personal Access Token

Go to GitHub → Settings → Developer Settings → Personal Access Tokens and generate a fine-grained token with repo scope (read & write access to your repositories). Copy the token — you will only see it once.

Edit the Claude Desktop config

Open the Claude Desktop configuration file for your platform:

# macOS
~/Library/Application Support/Claude/claude_desktop_config.json

# Windows
%APPDATA%\Claude\claude_desktop_config.json

Add the GitHub MCP server block. If the file already has other MCP servers, add a new entry inside mcpServers:

# claude_desktop_config.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_your_token_here"
      }
    }
  }
}

Save the file, then fully quit and relaunch Claude Desktop. The MCP server starts as a child process — changes to the config file are only picked up on launch.

Verify the connection

In a new Claude Desktop conversation, ask something like “List the files in my repo username/repo-name”. If Claude returns a file listing rather than an error, the MCP connection is live. If you see errors, the most common causes are a stale Gemfile.lock, a missing repo scope on the token, or npx not being available in your system PATH.

Step 2 — Set up a Jekyll GitHub Pages site

GitHub Pages natively builds Jekyll sites on every push to main. The minimal viable setup is three files.

Choose a theme

GitHub Pages supports a small set of officially sanctioned themes (Minimal, Slate, Cayman, etc.) via the remote_theme config key, plus any theme hosted on GitHub via jekyll-remote-theme. The cleanest setup for customisation is the Minimal theme — it provides a clean single-column layout with a left sidebar, minimal default styling, and easy CSS overrides.

remote_theme

pages-themes/minimal

Single-column, no opinionated sidebar. Best for a personal site where you control the full layout.

✓ Recommended for custom designs
remote_theme

pages-themes/slate

Dark background, two-column layout with a left sidebar. Good starting point for a technical blog.

⚠ Harder to override background colours

The three required files

Ask Claude to push these three files to get a working site in under a minute:

# _config.yml — site metadata and theme
title: "Your Name"
description: "Your tagline"
remote_theme: pages-themes/minimal@v0.2.0
plugins:
  - jekyll-feed
  - jekyll-remote-theme

# Gemfile — Ruby dependencies
source "https://rubygems.org"
gem "github-pages", group: :jekyll_plugins
group :jekyll_plugins do
  gem "jekyll-remote-theme"
end

# index.markdown — homepage content
---
layout: default
title: Your Site Title
---
Hello world.

A critical gotcha: never commit a stale Gemfile.lock. If one exists from a previous local setup, it will pin old gem versions that conflict with github-pages. Delete it or let GitHub Pages regenerate it on build.

Step 3 — Override the theme CSS

Every GitHub Pages theme loads its styles from the gem. The correct override path is assets/css/style.scss — Jekyll looks there first and merges it with the theme. The file must start with Jekyll front matter (two triple-dashed lines), then import the theme, then add your custom rules:

# assets/css/style.scss
---
---

@import "{{ site.theme }}";

# your overrides below — these load after the theme
body { background: #1a1a1a !important; }
a { color: #4ecdc4 !important; }

For layout-level elements like header, section, and footer, you often need !important because the theme sets these with high-specificity selectors. The most reliable approach is to combine SCSS variable overrides before the import line with explicit CSS overrides after it.

Step 4 — Override the layout

For deeper structural changes — a custom profile sidebar, different navigation, or a header with social links — create _layouts/default.html in your repo. Jekyll checks your repo’s _layouts directory before the theme’s, so your file wins automatically.

The minimal structure that works with Minimal theme is:

# _layouts/default.html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <link rel="stylesheet" href="{{ '/assets/css/style.css' | relative_url }}">
  {% include head-custom.html %}
</head>
<body>
  <header>...your profile sidebar...</header>
  <section>{{ content }}</section>
</body>
</html>

You can pull any site variable from _config.yml directly into this template — {{ site.title }}, {{ site.description }}, {{ site.github_username }} — making the header data-driven rather than hardcoded.

Step 5 — Push a blog post

Blog posts for a GitHub Pages site can be either Jekyll _posts markdown files (which get auto-listed by the theme) or standalone HTML files in a blog/ folder. For full design control — custom hero sections, sidebar cheatsheets, pull quotes — the standalone HTML approach gives you complete freedom with no theme interference.

Standalone HTML blog posts

Keep a template file at blog/_template.html with all your design tokens, component CSS, and placeholder markup. When you want a new post, tell Claude: “Create a new blog post following blog/_template.html about [topic] and push it.” Claude reads the template, writes the content, and pushes the file in one step.

Update the index manually

Since standalone HTML posts are not Jekyll posts, they won’t appear in site.posts. Maintain a manual post list in index.markdown using a consistent card component. Each card gets a tag, a date, a title link, a description, and a “Read the post” link. Ask Claude to add the new entry to the index in the same commit as the blog post.

Lessons learned — common pitfalls

ProblemRoot causeFix
Theme not applying Using theme: instead of remote_theme: for non-natively-supported themes Switch to remote_theme: pages-themes/minimal@v0.2.0 + add jekyll-remote-theme plugin
Build fails silently Stale Gemfile.lock pinning old Jekyll version Delete Gemfile.lock or regenerate it against github-pages gem
CSS overrides ignored Inline <style> in markdown loads after theme but misses structural elements Move all overrides to assets/css/style.scss with !important on layout selectors
Mobile sidebar overlap position: sticky escaping grid flow on narrow viewports Add @media (max-width: 780px) { .sidebar { display: none; } }
Blog layout not home Minimal theme has no home layout; that is Minima-specific Use layout: default in all page front matter

The workflow in practice

Once MCP is configured, the full loop — from idea to live page — collapses to a single conversation. You describe what you want; Claude reads the current state of your repo, writes the code, and pushes the commit. You refresh GitHub Pages one to two minutes later and see the result.

“The conversation becomes the version history. Your intent is the commit message.”

This is especially powerful for iterative UI work. Instead of switching between your editor, terminal, and browser, you describe the problem (“the sidebar overlaps content on mobile”) and Claude diagnoses, fixes, and pushes in one step. The feedback loop that used to take minutes per iteration shrinks to seconds.

For anyone building a technical blog or personal site, connecting Claude to GitHub via MCP is one of the highest-leverage setups available today. The one-time cost of configuration pays back immediately — and every future post or redesign becomes a conversation rather than a task.
