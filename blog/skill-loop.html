The problem with how most people build skills

You build a Claude skill. You run it a couple of times. The outputs look decent. You ship it and move on.

That was the only option. Until now.

Skills 1.0 gave you a SKILL.md file and told you to write good instructions. There was no way to measure whether those instructions were actually working — whether your skill was making outputs better or just adding noise. You were flying blind, optimizing by feel, crossing your fingers every time Claude loaded your skill.

"You go from a skill you hope works to a skill you've actually proven works."

Skills 2.0 changes that. It introduces structured evaluation, blind A/B benchmarking, and trigger optimization — transforming skill-building from guesswork into a feedback loop you can trust.

skill-loop is a guide skill that walks you through this entire methodology, step by step — whether you're creating a skill from scratch or auditing one that's been sitting untouched for months.

Two types of skills. One methodology.

Before building anything, you need to know what kind of skill you're building. The type shapes everything — how you write it, how long it stays useful, how aggressively you test it.

Capability Uplift

Teaches Claude something new

Your brand voice. A proprietary checklist. Industry-specific knowledge Claude doesn't have by default.

⚠ Expires as Claude improves
Encoded Preference

Documents your exact workflow

Not what to do — but where to look, what order, what format. Your process, locked in.

✓ Durable long-term

This distinction matters more than it seems. Capability Uplift skills have an expiration date — as Claude's base model improves, they can actually start holding Claude back. Encoded Preference skills are more durable, because they're not teaching Claude how to write — they're telling Claude your specific way to write.

The skill-loop methodology

The core methodology has one rule: always close the loop. Every change you make to a skill should be followed by a test, and every test should either confirm progress or tell you exactly what to fix next.

Path A — Building a new skill

Start by capturing intent before writing a single line of SKILL.md. Answer three questions:

What should this skill enable Claude to do? Be specific. "Write landing pages in my brand voice" beats "write better content" every time.

When should it trigger? What words or contexts should load the skill?

What does good output look like? Format, tone, structure, length — define it before you need to judge it.

Then write your SKILL.md. The body is straightforward: instructions, output format, examples. But the description field is where most skills fail silently.

The description is the trigger mechanism. Claude reads it to decide whether to load your skill at all. If it's too vague, the skill never fires. If it's too narrow, it misses the edge cases. The skill-loop guide teaches you to write descriptions that are deliberately "pushy" — Claude tends to undertrigger, so you need to compensate.

# ❌ Too vague — undertriggers
description: Helps write blog posts

# ✅ Pushy and specific — fires when it should
description: Writes blog posts in [your style]. Use whenever
             the user asks to write, draft, or create any
             long-form content — even without saying "blog post."

Path B — Improving an existing skill

If you have a skill that feels off, inconsistent, or was written before Skills 2.0 — don't touch it yet. Run evals first to get a baseline score. Even a low score (3 out of 9) is valuable: it tells you where you are, so you can prove progress later.

Then ask the most important question most skill builders never ask: is this skill still earning its place?

A/B benchmarking — the question nobody asks

Say you built a skill six months ago to help Claude write landing pages. At the time, Claude needed detailed instructions to do it well. Then Anthropic released a smarter model. Now Claude is already great at landing pages — but your old skill is still loaded, still forcing it through a rigid set of steps written for a less capable model.

Your skill is actively making outputs worse. And you'd never know without testing it.

A/B benchmarking runs the same prompts twice — once with your skill, once without — and has a blind comparator agent judge both outputs without knowing which is which. No bias. Just a result.

When to use skill-loop

Use it when:

— You want to build a new skill but aren't sure where to start
— You have an existing skill that "feels off" but you can't pinpoint why
— A Claude model update has shipped and you want to check if your skills still hold up
— You want to know whether a skill is actually helping — or just adding complexity
— Your skill fires at the wrong times, or doesn't fire when it should

Key commands to know

# Create a new skill
"Create a skill that helps me [describe task]"

# Test an existing skill
"Run evals on my [skill name] skill"

# Compare to raw Claude
"Benchmark my [skill name] against raw Claude"

# Fix a specific failure
"Update the skill to fix [specific problem]"

# Optimize when it triggers
"Optimize the description of my [skill name] skill"

The shift that changes everything

Skills 1.0 was a template. You wrote instructions and hoped Claude followed them. Skills 2.0 is a feedback loop — you test, you measure, you prove improvement, you repeat.

The developers who run evals will build better skills than those who don't. That's not an opinion — it's just math. One reported improving a skill's success rate from 67% to 94% in two optimization cycles, not by guessing better, but by testing systematically.

skill-loop exists to make that process accessible whether you're a first-time skill author or someone with a library of skills that's never been properly tested. Install it, run it, trust the loop.

Maintenance rule

If you haven't re-evaluated a skill in 3 months, treat it as unverified. Run an A/B benchmark after every major Claude model update.
