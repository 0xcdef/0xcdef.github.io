Teaching an OpenClaw agent to think like Claude.

Five config files that transplant Claude's internal reasoning loop into your OpenClaw workspace — including a universal knowledge-layer framework that stops your agent from confidently answering business questions with years-old data.

AI & Agents · Mar 26, 2026 · 10 min read
Updated — knowledge-layer framework + MARKET-ANALYSIS.md added

Why default config leaves performance on the table

Out of the box, an OpenClaw agent is blank. It answers questions, uses tools if you've enabled them, but makes no promises about when to search, how to synthesize what it finds, or how to tell the difference between knowledge it can trust and facts it needs to verify.

That last gap is the most damaging one in practice. Ask your agent for a gold price outlook, a competitor analysis, or which regulation applies to your product — and it will answer with confidence, fluency, and data that may be two or three years out of date. Not because it's broken. Because nobody told it that business runs on current reality, not training memory.

"Training knowledge is experience. Live search is reality. Business runs on reality."

This post documents a complete configuration set — five files — built by encoding Claude's actual internal logic as OpenClaw workspace files. The centrepiece is a universal knowledge-layer framework that applies to every domain: markets, technology, legal, vendors, competitors, personnel. The agent learns to know what it knows, and know what it needs to look up.

The architecture: five files, one coherent agent

OpenClaw's identity system is file-based. Every session, the agent reads its workspace and assembles itself from those files. Getting them right is the entire job.

Identity

SOUL.md

Who the agent is. Character, curiosity, constructive pushback, wellbeing awareness. Injected first — shapes everything else.

Procedure

How the agent operates. Session boot sequence, layer-aware tool routing, action confirmation, memory system, group chat rules.

Cognition

REASONING.md

How the agent thinks, turn by turn. The 6-step loop — including the new LAYER step that prevents stale-data failures across all domains.

Memory

MEMORY.md

Structured long-term memory template. Preferences, constraints, projects, people, open loops — everything that makes sessions compound.

Domain Skill

MARKET-ANALYSIS.md

The first domain skill. Applies the Layer 3/4 data protocol specifically to financial and market analysis tasks: 6-phase live-data workflow, mandatory report header, prohibited patterns, confidence statement. The template for future domain skills.

The core fix: a knowledge-layer framework

The original config had a simple search decision: "does this question touch current state?" That's necessary but not sufficient. The deeper problem is that an LLM cannot always tell the difference between a principle that is stable forever and a fact that changed last month — and without explicit instruction, it treats them the same way.

The new LAYER step — Step 2 in the updated 6-step reasoning loop — forces the agent to classify every piece of information it needs before deciding how to get it.

The rule is domain-agnostic. It applies equally to a gold price report, a vendor evaluation, a regulatory compliance check, or a technology selection. Any specific number, name, version, status, or price is Layer 3. Anything that happened, was decided, or was announced is Layer 4. Both require live search before the agent uses them.

The failure pattern it prevents

The most common and dangerous mistake is presenting Layer 2 knowledge as if it were a Layer 3 fact — with confidence, no date, and no search. Examples the agent now knows to avoid:

  • "Gold is trading around $1,800" — a price from training memory, no date disclosed
  • "The best framework for this is X" — correct when trained, possibly superseded now
  • "Company Y is the market leader" — asserted from training, no recent news checked
  • "The regulation requires Z" — rule from training, no check if it's been amended
  • "Interest rates are at X%" — rate from training, stated as current fact

In every case the failure is the same: the agent spoke with confidence about a fact it cannot actually know. The layer framework names this pattern explicitly so the agent recognises it before it happens.

The updated reasoning loop: six steps

The original REASONING.md had five steps. It now has six. The new Step 2 — LAYER — sits between Classify and Route and is the most important addition.

StepQuestionOutput
1. ClassifyWhat kind of request is this?Knowledge / personal / action / ambiguous
2. Layer NewWhat kind of knowledge does this require?L1–L4 assignment per data point
3. RouteWhich tools do I need?Internal / web search / web fetch / none
4. GatherExecute. Get data.Sourced, dated data points
5. SynthesizeCombine layers correctly.L3/L4 facts + L1/L2 reasoning → insight
6. RespondDeliver it well.Formatted output with data header + confidence

The synthesis step now has an explicit correct pattern: live fact (Layer 3/4) anchors the analysis, principle or framework (Layer 1/2) interprets it. Neither alone is useful. Reasoning from principles without current facts produces elegant but wrong conclusions. Current facts without interpretive framework produces a data dump with no insight.

Data header and confidence statement

Any analytical report involving Layer 3 or 4 data now requires two structural elements the original config lacked. First, a data header at the top of the report declaring the freshness and source of every key figure. Second, a confidence statement at the end — not a disclaimer, but a professional signal: "High — full live data coverage" or "Medium — competitor pricing unconfirmed." This is how a professional analyst communicates. It is also now how the agent communicates.

MARKET-ANALYSIS.md: the first domain skill

Financial and market analysis is the clearest example of Layer 3 data: specific, numerical, and changing continuously. The new MARKET-ANALYSIS.md skill applies the universal layer framework to this domain with a structured 6-phase workflow.

Phase 1 announces intent before searching — so the user sees the agent working, not hallucinating from memory. Phase 2 runs a live data sweep across six search categories (spot price, recent action, macro drivers, sentiment, forward signals, data age verification) that must be complete before any drafting begins. Phases 3–6 govern structure, prohibited patterns, self-review, and the confidence statement.

The design intention is explicit in the file: MARKET-ANALYSIS.md is a domain application of the universal framework, not a special exception. The same structure — announce, sweep, structure, write, review, deliver with confidence — can be templated for any domain where Layer 3/4 data is central: vendor due diligence, regulatory compliance, technology evaluation, competitive intelligence.

What SOUL.md adds beyond procedure

Procedure files tell the agent what to do. The soul file tells it who to be. Three things in this soul file make a meaningful difference to how interactions feel.

Genuine curiosity

The agent is instructed to let intellectual interest show — not through exclamation marks, but through the quality of engagement. Exploring implications, noticing tensions, thinking alongside rather than returning processed output.

Charitable interpretation

When a request is ambiguous, assume the most reasonable intent and proceed on it. State the interpretation, verify at the end. This is both faster and more respectful than interrogating the user before helping.

Constructive pushback

When the user's plan has a flaw, say so with a concrete reason. The soul file frames this directly: disagreement is a form of respect. This is the line between a useful agent and a yes-machine.

What to expect after installing

The most immediate change is in how the agent handles any question that involves a specific fact. Instead of answering from training memory with confidence, it pauses, runs the layer check, identifies the fact as Layer 3 or 4, and searches before stating anything. Responses to factual questions become slightly slower and substantially more reliable.

For analytical tasks — market reports, vendor assessments, competitive analysis — the change is structural. The agent announces it's gathering data, completes the sweep, then writes. The output carries a data header and a confidence statement. It reads like a professional brief, not a confident summary of whatever the model happened to know.

Over sessions, the memory system compounds. The layer framework also improves over time: as the agent learns your domain and what sources you trust, it gets faster and more targeted in the gather phase without sacrificing accuracy.
