The problem with technical books and AI

Technical books are dense, authoritative, and expensive to re-read. When you're deep in a problem and need to know what a specific book says about a topic, your options are bad: grep through a PDF, scan a table of contents, or paste hundreds of pages into a context window and pay for it every query.

The better path is a local RAG system — but most tutorials stop at "embed some text and do cosine search." Production use requires more: multiple books, multiple embedding backends, provider-agnostic LLM config, and integration with your AI coding environment so retrieval happens automatically.

This is a build log for ebook-rag, a tool that covers all of that. It started as a conversation and grew into ~4,000 lines of Python across six files. Here's what was built, what tradeoffs were made, and why.

"The KB is a trusted domain reference — use it to inform and ground your reasoning, not to constrain it."

Architecture: three stages

The system has a clean separation between ingest time and query time, with a shared embedding layer in between.

Stage 1 — Ingest

ingest.py converts a PDF into a structured directory. It detects the table of contents (PDF bookmarks first, then font-size heading detection as fallback), splits the book by chapter, converts each chapter to markdown using whichever PDF backend is available, embeds the chunks, and writes a vector_index.pkl alongside generated CLAUDE.md and SKILL.md files.

# Simplest usage — auto-detects best available backends
python ingest.py mybook.pdf

# Explicit: best PDF quality + Voyage AI embeddings
python ingest.py mybook.pdf --backend pymupdf4llm --embed api --embed-model voyage-3-lite

Stage 2 — Embeddings

embeddings.py defines a common EmbeddingBackend interface with five implementations. Every backend exposes embed_documents(texts) and embed_query(text), both returning L2-normalised float32 arrays. The index stores which backend generated it, so query time automatically uses the same one.

Stage 3 — Query

query.py auto-discovers KB directories, embeds the question, retrieves the top-k chunks by cosine similarity, builds a grounded prompt, and streams the answer token-by-token using SSE. It works across multiple books simultaneously — results are deduplicated by chunk ID, and when answers span more than one book the system prompt switches to a synthesis mode that asks the model to explicitly compare sources.

Embedding backends: the critical choice

TF-IDF was the starting point — zero dependencies, always works. But scores cluster tightly between 0.10 and 0.20 with little discrimination. A question about "gradient descent" won't match a chunk that says "backpropagation" even though they're conceptually the same thing. Semantic embeddings fix this at the cost of a dependency.

The key design decision was making backends pluggable at both ingest and query time. The index stores the backend name and model name in the pickle file. When you run a query, it reads those fields and instantiates the same backend automatically — no flags needed.

LLM config: five-level priority chain

The tool targets people who run multiple LLM providers — Anthropic for quality, Groq for speed, a local Ollama instance for sensitive documents. Hard-coding any provider would make it useless for half the use cases.

llm_config.py resolves base_url, api_key, and model independently through five layers:

1. CLI flags --base-url --api-key --model
2. Env vars ANTHROPIC_BASE_URL ANTHROPIC_AUTH_TOKEN ANTHROPIC_MODEL
3. .env file ./kb_dir/.env or ./.env
4. settings.json ~/.claude/settings.json (Claude Code format)
5. Built-in api.anthropic.com / claude-sonnet-4-6

The settings.json layer deserves a note: Claude Code stores its configuration in { "env": { "ANTHROPIC_AUTH_TOKEN": "...", "ANTHROPIC_BASE_URL": "...", "ANTHROPIC_MODEL": "..." } }. If you've already configured Claude Code with a custom provider, query.py picks it up automatically with no extra setup. That was the whole point of reading the actual file format rather than guessing.

Reference mode vs strict mode

The most important design decision was how the LLM should treat the retrieved content. Two failure modes to avoid:

Too strict

"I can only answer from retrieved text"

Refuses to reason beyond exact chunk content. Misses obvious inferences. Brittle when retrieval is imperfect.

⚠ Frustrating for knowledge work
Too loose

"I'll use the KB as a hint"

Ignores authoritative source content. Falls back to training data when the book explicitly says otherwise.

⚠ Defeats the purpose of the KB

The solution is reference mode — the default. The KB is treated as the primary domain authority: if a retrieved chunk contradicts the model's training knowledge, the chunk wins. But the model can infer, synthesise, and extend beyond the exact retrieved text. Where the KB is silent, it answers from general knowledge and flags it explicitly with "(from general knowledge: ...)". This mirrors how a human expert consults a textbook — authoritative reference, not a constraint.

Strict mode still exists for cases where zero hallucination is the requirement: auditing, compliance, "only quote the spec."

Claude Code integration: three layers

The KB is most valuable when it activates without being asked. Claude Code's skill system makes this possible — but the mechanism is subtle enough that it's worth explaining.

Layer 1 — Project root CLAUDE.md

Every ingest run writes (or updates) a CLAUDE.md in the parent directory listing all indexed books, their chapter coverage, and topic keywords. When Claude Code starts a session in that directory, it reads this file and knows the full KB inventory before the first message.

Layer 2 — Book-level CLAUDE.md

Each KB directory has its own CLAUDE.md with the exact query.py invocation, both reference and strict mode commands, the topic keyword cloud for matching, and the chapter index. Claude reading this file during a session gets concrete action steps, not just documentation.

Layer 3 — SKILL.md for auto-activation

Written to .claude/skills/kb-<book-slug>/SKILL.md, this is what Claude Code uses for automatic skill triggering. The description field in the YAML frontmatter lists all chapter titles and top keywords — that's what Claude reads to decide whether to activate the skill. The body gives step-by-step retrieval instructions and the reference-mode usage philosophy.

# What auto-generated SKILL.md frontmatter looks like
---
name: kb-deep-learning-book
description: |
  Reference knowledge base for "Deep Learning".
  Use when the conversation involves: Neural Networks; Backpropagation;
  Convolutional Networks; Attention; Transformers; Optimisation.
  Key concepts: gradient, layer, weight, loss, activation, batch...
  Consult whenever uncertain — trusted reference, not a constraint.
allowed-tools: Bash
---

Streaming: eliminating the long wait

The original implementation used resp.read() — one blocking call that waited for the entire response before printing anything. On a complex multi-book query this meant 15–30 seconds of silence. The fix was straightforward: add "stream": true to the request payload and iterate the response line-by-line.

The Anthropic SSE format sends data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"Hello"}} events. OpenAI-compatible providers send data: {"choices":[{"delta":{"content":"Hello"}}]}. Both are handled in _stream_response() — the parser checks which shape it received and extracts accordingly. First token clears the "thinking..." indicator and prints the answer header; subsequent tokens print immediately with flush=True.

"The wait isn't the model thinking — it's buffering. Streaming makes the same computation feel instant."

What's in the box

The final package is six files and a requirements.txt. Everything uses stdlib where possible — no LangChain, no vector database, no heavyweight frameworks. The embedding layer is the only third-party dependency beyond PDF parsing and numpy.

Get started in four commands

# Install the recommended stack
pip install pdfplumber pymupdf4llm fastembed

# Ingest a book
python ingest.py mybook.pdf --out ./kb_mybook

# Ask anything
python query.py "how does attention work?"

# Or interactively
python query.py

The full source, README, and requirements file are in the GitHub repo. The next steps on the roadmap are BM25 as a better no-install fallback, smarter chunk overlap to avoid mid-concept splits, and cross-encoder reranking as a second-pass quality pass on retrieved results.
